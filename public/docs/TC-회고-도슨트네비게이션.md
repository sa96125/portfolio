CARD-2 : 도슨트 및 추천 경로 네비게이션 구축
(메트릭퍼스트)
비전공 영역인 공간 데이터를 자체 구축하고, AI 도슨트 네비게이션 시스템을 설계한 경험

- 15만평 사유지 GIS DB 자체 구축
- 외주 견적 2억+ 절감 (위치수집 8천만원 포함)
- 일평균 400~600명 사용자 안정 운영

(역할)

- 백엔드 API 개발
- GIS 데이터 처리 파이프라인 설계 (PostGIS + pgRouting),
- AI 기반 추천 경로 / 도슨트 해설 프롬프트 설계
- QFieldCloud 자체 호스팅 인프라 구축

(배경)
지류 지도만으로는 방향, 거리, 소요시간을 직관적으로 파악할 수 없다는 고객 불만이 누적되었습니다. LG 화담숲 개발사로부터 받은 외주 견적이 2억 이상 — 그중 위치수집 견적만 8천만원이었고 GIS는 회사 누구도 경험이 없는 비전공 영역이었기에 전문 영역으로 인식되어 사업 중단 위기.

(과정)
Step1. 외주 견적 검토 → 위치수집 영역만 8천만원
Step2. 오픈소스 GIS 스택 조사 → QGIS 채택 (커뮤니티 규모 + 무료)
Step3. 현장 수집 환경 검토 → 산속 통신 단절 환경에서 오프라인 수집 가능한 QField 발견
Step4. 사용자(직원) 운영 부담 최소화 → 로그인만으로 동기화되는 QFieldCloud(QFC) 자체 호스팅으로 정착
Step5. QFC 공식 레포 클론 후 마이크로서비스 전체 직접 셋팅 (단순 docker-compose 아닌 다중 마이크로 서비스 조합)
Step6. 외장 GPS 수신기 + QField 블루투스 연결로 cm 단위 정밀도 확보
Step7. 측정 데이터 자동 처리 파이프라인 구축 (PostGIS + pgRouting, 트리거 기반 자동화)
Step8. 사유원 도메인 온톨로지 설계 → AI 추천 경로/도슨트 해설 초안 자동 생성

(문제해결)
첫째, 휴대폰 GPS 정밀도 한계 + 산속 신호 약함
도슨트 네비게이션은 cm 단위 정밀도가 요구되는데
휴대폰 내장 GPS는 5~10m 오차에 산속에서는 신호 끊김.

직원 수집 단계: 외장 GPS 수신기를 QField에 블루투스로
연결, cm 단위 좌표 + 고도까지 수집.

사용자 단계: 위치 캐싱 + 마지막 알려진 위치 활용으로
신호 일시 끊김에도 안내 연속성 확보.

둘째, 입장 시간대 트래픽 집중 + 대량 데이터 처리
일평균 400~600명, 입장 시간대(오전 10시) 동시 접속 50+,
수백 개 POI, 다국어 음성 수천 개를 효율적으로 처리해야 함.

- 위치 추적/POI 진입 감지: 클라이언트 처리 → 서버 부하 최소화
- 도슨트 음성/경로: 정적 파일은 CDN/스토리지 직접 제공
- 공간 검색: PostGIS GiST 인덱스로 가속
- 자주 조회되는 경로: Redis 캐싱
- 백엔드: 가벼운 API만 담당

"실시간 처리와 정적 자원을 명확히 분리"한 설계로
단순한 인프라로 안정 운영 확보.

셋째, 4개국어 도슨트 운영의 복잡도
4개국어 × POI 수 = 수천 개 음성 파일 관리.
어르신 방문객 + 외국인 통역 인력 부담 동시 해결 필요.

AI 해설 초안 자동 생성 → 현업 검수 → ElevenLabs 4개국어
사전 합성 구조. 파일 명명 규칙 표준화로 관리자 도구에서
일괄 갱신.

"AI 즉시 응답"이 아닌 "검수 후 사전 합성" 구조로
어르신 방문객 대상 서비스의 안정성 확보.

2-1) QFieldCloud 자체 호스팅
공식 레포 클론 후 마이크로서비스 전체를 직접 구성.
단일 docker-compose가 아닌, 인증/스토리지/PostGIS/Worker 등 다중 서비스 조합.

2-2) 온톨로지
사유원의 POI(정자, 전망대, 수목 등), 코스, 계절성, 난이도, 인접 관계를 구조화.
복잡한 질의는 없는 도메인이므로 GraphDB 도입 없이 시스템 프롬프트 + 관계형 모델링으로 운영. "필요 이상으로 무거운 기술 도입을 피하고, 도메인 복잡도에 맞는 수준으로 설계"한 의사결정.

(2-3) 측정 데이터 처리 파이프라인

원본 보존 + 단계별 정제 구조:
gps_point (원본)
→ gps_track_optimized (자동 가공)
→ gps_path (관리자 확정 경로)
→ gps_map (통합 라우팅 그래프)

자동 처리:

- 노이즈 감지 (GPS 정확도, 불가능한 점프)
- 트랙 자동 분리 (위치 점프 시 라인 분리)
- 단순화 (Douglas-Peucker) + 평탄화 (Chaikin)
- PostgreSQL 트리거로 동기화 시점 자동 실행

수동 처리 (의도적 제한):

- 사유원의 세분화된 길 구조 + GPS 정확도 한계로
  자동 통합 시 데이터 손실 위험
- "기계가 명백히 잘하는 것만 자동화"
- 노드 등록, 경로 통합은 모두 관리자 판단

데이터 무결성:

- 원본 gps_point 절대 수정 X
- 단계별 source 추적 가능
- 문제 시 원본부터 재가공 가능
