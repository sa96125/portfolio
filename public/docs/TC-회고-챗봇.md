STAR : RAG 기반 챗봇 구축
(메트릭퍼스트)
10만건의 문서 지식을 주입한 AI 챗봇 개발 경험

- 검색정확도 95%
- AI 개발 운영비 64% 절감

(역할)

- 챗봇 디자인/프론트 개발
- LLM 파인튜닝, 임베딩
- 인프라 관리 (GCP, Elice)

(배경)
사내에서는 "이 자료 있나요?", "이런 절차 있나요?" 같은 문의가 부서 간에 반복적으로 발생하고 있었습니다. 자료가 그룹웨어에 누적되어 있어도 키워드 검색만으로는 원하는 정보를 찾기 어려웠고, 결국 특정 담당자를 거쳐야만 정보가 흐르는 지식 병목 현상이 부서마다 나타나고 있었습니다.
마침, 저희 팀은 AI 도입 시점을 고민하고 있었는데, 저는 이번 기회에 이 일을 AI로 대체할 수 있지않을까? ChatGPT를 초기부터 써온 입장에서 자료를 읽고 분석해 답하는 일은 AI가 가장 잘하는 영역이라는 확신이 있었습니다. 무엇보다, 새로운 기술에 부담을 느끼던 분들도 평소 쓰던 그룹웨어 안에서 자연스럽게 AI를 만날 수 있게 하고 싶었습니다. 단순 검색을 넘어 자료의 맥락까지 짚어 의사결정을 거들 수 있다면, 신입사원부터 대표까지 직급과 부서를 가로질러 모든 구성원의 생산성을 동시에 끌어올릴 수 있는 일이었습니다.
정보가 가장 많이 모이고 가장 잘 활용될 수 있는 그룹웨어에 챗봇을 연동하자고 제안했고, 그렇게 시작되었습니다.

(과정)
Step1. 허깅페이스 오픈소스 테스트 Qwen3
└─ Context - 사내 보안 요건상 외부 API(GPT/Claude) 의존 불가 - 한정된 자원: 단일 GPU 환경, 보수적 분위기 → 운영 가능한 모델 후보 좁음 - HuggingFace 생태계 기반 오픈소스 모델 비교 테스트(한국어)
└─ Decision - HuggingFace 오픈소스 모델 후보군 비교 테스트 후, Qwen3-Instruct 채택
· 파라메터크기(7~32B)/변형모델(Base/Instruct 등)/양자화호환성 고려  
 · 후보 간 비교는 공개 벤치마크에 의존하지 않고 휴먼 검증
· 사내 도메인 샘플 질의/응답으로 직접 입출력 비교
· "벤치마크 점수가 높은 모델"이 아니라"우리 데이터에서 입출력이 좋은 모델"을 선택
· 공개 벤치마크는 일반 도메인 + 깨끗한 데이터 기준
· 사내 문서는 행정 문체 + 약어 + 표/리스트 혼재 → 벤치마크와 분포가 다름
└─ Consequences + 외부 API 비용 0, 데이터 외부 유출 0 + 단일 GPU 양자화 서빙 가능성 확인 + 공개 벤치마크 의존 X → 도메인 적합성 직접 검증 + RAG에 적합한 모델 변형 선택 (Instruct) - 베이스 모델은 일반 지식만 보유, 사내 도메인 모름 - 약어/프로세스 질의에 그럴듯한 거짓말 (도메인 지식 부재 한계) - 휴먼 검증은 정성 평가라 재현성 약함 (Step6에서 평가셋으로 정량화)
→ Next: 도메인 지식을 어떻게 주입할 것인가?
→ Next: 개발 비용을 최소화하면서 최대 효율을 낼 방법은?

Step2. 파인튜닝 및 인프라 구축
└─ Context - 모델의 지식과 사고능력을 활용하고, 도메인 지식만 학습 - 월 30만원 이하로 GPU 대여 → 풀 파인튜닝 불가 - LoRA/qLoRA 계열은 어댑터만 학습하니 보존된다는 당시 통념
└─ Decision - QLoRA + 4bit 양자화 + unsloth 조합 시도
· QLoRA 하이퍼파라미터 튜닝: rank(r), alpha, dropout 테스트
· 학습 데이터 품질: 사내 문서 정제, instruction 형식 가공
· A100-40G에서 32B - 4bit 양자화 파인튜닝 가능
· Unsloth: 학습 속도/메모리 최적화 - GCP spot 요금제 사용하여 비용 절감
· 인스턴스가 언제든지 중단될 수 있어 최대 90% 비용 절감 가능
· 관리자 수동 재실행 불편, 중단 감지하여 자동 재실행 배치파일로 대응
└─ Consequences + 단일 GPU 32B 파인튜닝 가능 + 사내 챗봇으로 사용하기에 32B도 사고력이 충분 - 재앙적 망각: 일반 지식/추론 능력 붕괴 - 할루시네이션 폭발: Step1 Instruct의 "모르겠다" 능력 상실 - 사내 문서 자주 갱신 → 재학습 비용 trade-off - Spot은 가용자원이 없을 경우, 재실행 자체 불가능 (개발용으로만 사용) - "LoRA 망각 회피"는 학습량 조건부, 통념의 한계 확인
Biderman et al.(2024) "LoRA Learns Less and Forgets Less" - 변수 통제 후에도 본질적 한계 → "모델을 가르치기"보다 "모델에게 자료를 주기"
→ Next: 자료를 외부에 두고 검색해서 LLM에 주입하는 구조는?

Step3. RAG 전환 (Qwen3 + BGE-M3 + VectorDB)
└─ Context - 사용자 쿼리로 자료를 검색한 후, 답변 생성 - 파인튜닝에 비해 검색 시간은 느리겠지만, 정확도를 높히는 것을 목표함.
└─ Decision - 한국어 RAG 스택 후보군 비교 테스트 후 채택
· 임베딩: BGE-M3 (한국어 임베딩 SOTA, dense + sparse 동시 지원)
· VectorDB: Qdrant (hybrid 검색 지원, 자체 호스팅 가능) - 프레임워크 Langchain 배제  
 · 한국에서의 인기와는 별개로 RAG 전용 자료는 LlamaIndex 쪽이 더 깊이 있음을 확인 - 사내 문서는 Pdf, Excel, docx, html + 표/리스트 혼재하여 마크다운으로 변환 후, 청킹(400/40)
└─ Consequences + 모델 손상 없음 → "모르겠다" 능력 + 일반 지식 그대로 - 문서 갯수가 많아질수록, 검색 정확도 감소(10만개 기준 10%미만)

첫째, 프롬프트를 살짝 바꿔도 결과가 달라져서, 기존에 잘나왔던 항목 테스트가 어려움.

둘째, 보수적인 분위기, 한정된 자원으로 인프라를 사용해서 성과를 내야했음.
GCP spot 정책, 최대 90%까지 a100-40G를 월 24만원정도로 60프로정도 비용절감 빌려서
개발용도로 사용, Qwen3 32B 양자화 모델을 vllm으로 서빙, unsloth로 파인튜닝 가능했음.
셋째, 좋은 모델을 사용하더라도 검색결과가 좋지는 않음.
카오스이론, 앙상블 예측에 영감을 받아서, 4채널 가중치 + Hitscore체계로 이어질수 있었음

1-1) 파인튜닝 경험
파인튜닝으로 도메인 지식은 들어갔는데, 모델의 기존 일반 지식이 같이 망가지는 현상이 나왔습니다. 학술적으로는 재앙적망각(Catastrophic Forgetting)이라고 부릅니다. 거기에 양자화로 인한 정밀도 손실이 겹쳐 모델이 바보가 됐습니다.

HuggingFace 포럼, Reddit r/LocalLLaMA에서 LoRA/QLoRA 망각 사례 다수
좁은 도메인 데이터로 충분히 학습시키면, LoRA 계열도 일반 능력 손실이 발생한다는 걸 직접 확인했습니다. 학계에서도 'LoRA Learns Less and Forgets Less'(2024) 같은 논문에서 LoRA의 망각 회피 효과는 학습량 조건부라는 게 정리되고 있습니다. 'LoRA니까 안전하다'는 건 반쪽짜리 진실이었습니다."
